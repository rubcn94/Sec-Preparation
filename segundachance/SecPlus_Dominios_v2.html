Security Controls

Security Controls Security Controls ▼

Salvaguardas o contramedidas para reducir riesgo. Tipos: preventivo, detective, correctivo, disuasorio, compensatorio, directivo. Categorías: técnico, administrativo, físico.

🧠 Controls → 'Preventivo para antes, Detective para durante, Correctivo para después'
✅ Para qué SÍ sirve
  • Clasificar un firewall (preventivo/técnico)
  • Responder preguntas de examen sobre qué control aplicar en un escenario
❌ Para qué NO sirve
  • Un control preventivo garantiza que el ataque no ocurra (solo reduce probabilidad)
  • Un solo control cubre todos los tipos
🎯 Patrón Sec+

Preguntas de escenario: 'cámara de seguridad' → detective+físico. 'MFA' → preventivo+técnico. 'política de uso aceptable' → directivo+administrativo. Trampa: confundir disuasorio con preventivo.

Defense in Depth Defense in Depth ▼

Estrategia de múltiples capas de controles de seguridad. Si una capa falla, la siguiente la detiene. Principio de capas redundantes.

🧠 Defense in Depth → 'Cebolla: cada capa protege si la exterior falla'
✅ Para qué SÍ sirve
  • Diseñar arquitecturas con firewall + IDS + endpoint + logs
  • Justificar por qué se necesitan controles redundantes
❌ Para qué NO sirve
  • No significa duplicar el mismo control, sino capas diferentes
  • No elimina la necesidad de parchear vulnerabilidades individuales
🎯 Patrón Sec+

Pregunta sobre diseño: 'además del firewall, ¿qué más?' → respuesta con capas adicionales (EDR, SIEM, segmentación). Trampa: pensar que un control muy bueno es suficiente.

Gap Analysis Gap Analysis ▼

Proceso de comparar el estado actual de seguridad con el estado objetivo o requerido. Identifica brechas entre controles existentes y requeridos.

🧠 Gap Analysis → 'Fotografia: dónde estás vs dónde debes estar'
✅ Para qué SÍ sirve
  • Preparar una auditoría de cumplimiento (NIST, ISO 27001)
  • Priorizar inversiones de seguridad identificando las brechas mayores
❌ Para qué NO sirve
  • No es una evaluación de vulnerabilidades técnicas (eso es vulnerability scanning)
  • No produce un plan de remediación por sí solo
🎯 Patrón Sec+

Enunciado con 'cumplimiento', 'estado actual vs requerido', 'framework' → Gap Analysis. Trampa: confundirlo con Vulnerability Assessment (técnico) vs Gap Analysis (estratégico/documental).

Change Management Change Management ▼

Proceso formal para gestionar cambios en sistemas IT. Incluye solicitud, aprobación (CAB), prueba, implementación y revisión post-cambio.

🧠 Change Management → 'Todo cambio necesita aprobación antes de tocar producción'
✅ Para qué SÍ sirve
  • Aprobar un parche de seguridad antes de desplegarlo en producción
  • Documentar cambios para auditorías y análisis forense post-incidente
❌ Para qué NO sirve
  • No es opcional aunque sea urgente (existe proceso de cambio de emergencia)
  • No garantiza que el cambio sea seguro, solo que está controlado
🎯 Patrón Sec+

Escenario de examen: 'sistema se cayó después de actualización no documentada' → falta de Change Management. Trampa: pensar que urgencia omite el proceso.

Fundamental Security Concepts

CIA Triad Confidentiality, Integrity, Availability ▼

Los tres pilares de la seguridad de la información. Confidencialidad: solo quienes deben acceder acceden. Integridad: datos no alterados sin autorización. Disponibilidad: sistemas accesibles cuando se necesitan.

🧠 CIA → 'Confidencial para ojos correctos, Íntegro sin alteraciones, Accesible cuando se necesita'
✅ Para qué SÍ sirve
  • Clasificar cualquier control o ataque en términos de qué pilar afecta
  • Ransomware afecta Disponibilidad e Integridad; espionaje afecta Confidencialidad
❌ Para qué NO sirve
  • CIA no es un framework de implementación, es un modelo conceptual
  • No-repudio es un concepto adicional, no está dentro de CIA
🎯 Patrón Sec+

Casi cualquier pregunta de examen puede responderse identificando qué pilar de CIA se ve afectado. Trampa: DDoS afecta A (no C), cifrado sin clave afecta C (no A).

AAA Authentication, Authorization, Accounting ▼

Framework de control de acceso. Autenticación: ¿quién eres? Autorización: ¿qué puedes hacer? Accounting: ¿qué hiciste? Implementado por RADIUS, TACACS+, Kerberos.

🧠 AAA → 'Quién eres, qué puedes, qué hiciste: el trio del acceso'
✅ Para qué SÍ sirve
  • Diseñar control de acceso a red (NAC) con RADIUS
  • Auditorías: el accounting registra qué hizo cada usuario
❌ Para qué NO sirve
  • Autenticación no implica autorización automática
  • AAA no cifra los datos del tráfico de red
🎯 Patrón Sec+

Pregunta: 'registrar quién accedió a qué recurso cuándo' → Accounting. 'verificar credenciales' → Authentication. 'definir qué puede hacer el usuario' → Authorization.

Zero Trust Zero Trust Architecture ▼

Modelo de seguridad que asume que ninguna entidad (interna o externa) es confiable por defecto. 'Never trust, always verify.' Requiere autenticación y autorización continuos.

🧠 Zero Trust → 'Nunca confíes, siempre verifica: ni la red interna es segura'
✅ Para qué SÍ sirve
  • Arquitectura para entornos híbridos/cloud donde el perímetro ya no existe
  • Acceso granular por identidad en lugar de por ubicación de red
❌ Para qué NO sirve
  • Zero Trust no elimina la necesidad de firewall o segmentación
  • No es un producto, es una arquitectura/filosofía
🎯 Patrón Sec+

Pregunta con 'empleados remotos', 'BYOD', 'nube', 'perímetro difuminado' → Zero Trust. Trampa: confundirlo con ZTNA (que es la implementación de acceso remoto de Zero Trust).

Non-repudiation Non-repudiation ▼

Garantía de que el emisor de un mensaje no puede negar haberlo enviado. Se implementa con firmas digitales. Combina autenticidad e integridad.

🧠 Non-repudiation → 'Firma digital: no puedes negar lo que firmaste'
✅ Para qué SÍ sirve
  • Transacciones financieras que deben ser irrefutables
  • Correos firmados digitalmente en contexto legal
❌ Para qué NO sirve
  • Cifrado solo no garantiza no-repudio (AES cifra, pero no firma)
  • No-repudio no garantiza confidencialidad por sí solo
🎯 Patrón Sec+

Pregunta: '¿qué garantiza que el remitente no pueda negar el mensaje?' → firma digital / non-repudiation. Trampa: confundirlo con autenticación (que solo verifica identidad en el momento).

Least Privilege Principle of Least Privilege ▼

Principio de seguridad: dar a cada usuario/proceso solo los permisos mínimos necesarios para su función. Reduce la superficie de ataque si la cuenta es comprometida.

🧠 Least Privilege → 'Solo lo que necesitas: ni un permiso más'
✅ Para qué SÍ sirve
  • Configurar cuentas de servicio con permisos mínimos
  • Revisión periódica de accesos para revocar lo innecesario (Access Review)
❌ Para qué NO sirve
  • No significa que todos los usuarios tengan los mismos permisos mínimos
  • No elimina la necesidad de autenticación fuerte
🎯 Patrón Sec+

Escenario: 'empleado puede acceder a datos de otros departamentos' → violación de Least Privilege. Solución → RBAC + revisión de accesos. Trampa: confundir con Need-to-know (similar pero aplicado a info clasificada).

Separation of Duties Separation of Duties ▼

Control preventivo: dividir tareas críticas entre múltiples personas para que ninguna pueda cometer fraude o error sin que otro lo detecte.

🧠 Separation of Duties → 'Dos llaves para el maletín nuclear: nadie actúa solo'
✅ Para qué SÍ sirve
  • Que quien autoriza pagos no sea quien los ejecuta
  • Que el desarrollador no pueda desplegar en producción directamente
❌ Para qué NO sirve
  • No protege contra colusión (dos personas que conspiran juntas)
  • No es lo mismo que Dual Control (que requiere dos personas simultáneas)
🎯 Patrón Sec+

Pregunta sobre fraude interno o control financiero → Separation of Duties. Trampa: confundir con Job Rotation (que también reduce fraude pero mediante rotación).

Change Management & Security Impact

Approval Process Change Management — Approval Process ▼

Proceso formal de aprobación antes de implementar cambios en sistemas IT. Incluye el Change Advisory Board (CAB) que evalúa riesgo, impacto, backout plan y ventana de mantenimiento.

🧠 Approval Process → "Ningún cambio llega a producción sin pasar por el CAB: aprobación obligatoria"
✅ Para qué SÍ sirve
  • Aprobar parches críticos de seguridad antes de desplegarlos en producción
  • Documentar ownership, stakeholders e impact analysis para cada cambio
❌ Para qué NO sirve
  • Un cambio urgente no omite el proceso — existe el Emergency Change para situaciones críticas
  • Aprobación no garantiza que el cambio sea seguro, solo que está controlado y documentado
🎯 Patrón Sec+

Escenario: "sistema caído tras actualización no documentada" → falta de Change Management/Approval Process. Trampa: incluso cambios pequeños necesitan aprobación; el backout plan es obligatorio.

Backout Plan Change Management — Backout Plan ▼

Procedimiento documentado para revertir un cambio fallido y restaurar el sistema al estado anterior. Obligatorio antes de implementar cualquier cambio en producción.

🧠 Backout Plan → "El paracaídas del cambio: si algo falla, sabes exactamente cómo volver atrás"
✅ Para qué SÍ sirve
  • Restaurar la versión anterior de una aplicación si el despliegue nuevo causa errores
  • Cumplimiento del proceso de Change Management: sin backout plan no se aprueba el cambio
❌ Para qué NO sirve
  • No todos los cambios son reversibles fácilmente (ej. migraciones de base de datos irrevocables)
  • Backout plan documentado ≠ backout plan probado; debe probarse en entorno de pre-producción
🎯 Patrón Sec+

Pregunta: "¿qué debe existir antes de implementar un cambio en producción?" → Backout Plan + Maintenance Window + Approval. Trampa: el backout plan debe estar listo ANTES del cambio, no improvizado si falla.

Version Control Version Control ▼

Sistema que rastrea cambios en código, configuraciones y documentación a lo largo del tiempo. Permite revertir a versiones anteriores, auditar quién cambió qué y cuándo. Ejemplos: Git, SVN.

🧠 Version Control → "La máquina del tiempo del código: cada commit es un punto al que puedes volver"
✅ Para qué SÍ sirve
  • Auditoría de cambios: quién modificó qué archivo de configuración y cuándo
  • Rollback rápido si un cambio introduce una vulnerabilidad o rompe la aplicación
❌ Para qué NO sirve
  • Version control no reemplaza backups — un repositorio corrupto o borrado pierde todo
  • No es solo para código: configuraciones, IaC, políticas también deben versionarse
🎯 Patrón Sec+

Pregunta: "control para rastrear cambios en configuraciones de sistemas y revertir si es necesario" → Version Control. Diferenciador: Version Control = histórico de cambios con autoría. Backup = copia de seguridad sin contexto de cambio.

Maintenance Window Maintenance Window ▼

Periodo de tiempo planificado y aprobado durante el cual se pueden realizar cambios en sistemas en producción, minimizando el impacto en usuarios. Suele ser fuera de horas pico.

🧠 Maintenance Window → "La franja horaria sagrada del cambio: las 2AM del domingo es tu mejor amiga"
✅ Para qué SÍ sirve
  • Aplicar parches críticos durante horas de baja actividad para minimizar impacto
  • Cumplir SLAs de disponibilidad coordinando cambios dentro de ventanas aprobadas
❌ Para qué NO sirve
  • Cambios fuera de maintenance window sin aprobación son violaciones del proceso de change management
  • La ventana no garantiza que el cambio no tenga impacto, solo que está en el momento menos disruptivo
🎯 Patrón Sec+

Escenario: "administrador aplicó parche un martes a las 2PM sin avisar" → violación de Maintenance Window. Trampa: emergencias tienen proceso de emergency change, no se omite la ventana sin autorización.

Cryptographic Solutions

AES Advanced Encryption Standard ▼

Cifrado simétrico de bloque (128/192/256 bits). Estándar NIST. Modos: CBC, GCM, CTR. AES-256 es el más seguro. Cifra datos, no intercambia claves.

🧠 AES → 'El candado simétrico universal: cifra datos, no intercambia claves'
✅ Para qué SÍ sirve
  • Cifrar discos y volúmenes en reposo (BitLocker, FileVault)
  • Cifrar datos dentro de TLS una vez establecida la sesión
❌ Para qué NO sirve
  • Intercambiar claves (eso es RSA o ECDHE)
  • Autenticar identidades o garantizar no repudio
🎯 Patrón Sec+

'Cifrado simétrico', 'datos en reposo', '10.000 laptops' → AES-256. Trampa: confundirlo con RSA (asimétrico, solo para claves).

AES-GCMTLSHSMKMS
RSA Rivest–Shamir–Adleman ▼

Algoritmo asimétrico basado en factorización de números primos. Clave pública cifra / clave privada descifra. Usado para intercambio de claves y firmas digitales.

🧠 RSA → 'Dos llaves: la pública cifra, la privada descifra. Lento pero seguro para claves'
✅ Para qué SÍ sirve
  • Intercambio de clave de sesión en TLS (antes de ECDHE)
  • Firmas digitales en certificados X.509
❌ Para qué NO sirve
  • Cifrar volúmenes de datos grandes (demasiado lento, usa AES para eso)
  • Reemplazar ECDHE en TLS 1.3 (ya no se usa RSA key exchange en TLS 1.3)
🎯 Patrón Sec+

Pregunta sobre 'intercambio de claves asimétrico' o 'PKI' → RSA. Trampa: RSA no garantiza Perfect Forward Secrecy (ECDHE sí). TLS 1.3 eliminó RSA key exchange.

ECC Elliptic Curve Cryptography ▼

Criptografía asimétrica basada en curvas elípticas. Claves más cortas para la misma seguridad que RSA. ECC-256 ≈ RSA-3072.

🧠 ECC → 'RSA compacto: misma seguridad, clave más pequeña, ideal para móviles'
✅ Para qué SÍ sirve
  • Certificados y firmas en dispositivos con recursos limitados (IoT, móvil)
  • TLS moderno (ECDHE para intercambio de claves con PFS)
❌ Para qué NO sirve
  • No reemplaza AES para cifrado de datos (sigue siendo asimétrico)
  • No es cuánticamente seguro (algoritmos post-cuánticos están en desarrollo)
🎯 Patrón Sec+

Pregunta: 'dispositivo con CPU limitada, necesita criptografía asimétrica' → ECC. Trampa: tamaño de clave ECC vs RSA confunde (ECC-256 no es igual que RSA-256).

ECDHE Elliptic Curve Diffie-Hellman Ephemeral ▼

Protocolo de intercambio de claves con ECC. La E de Ephemeral significa que cada sesión usa claves temporales → Perfect Forward Secrecy (PFS).

🧠 ECDHE → 'Intercambio de claves efímeras: cada sesión tiene su clave, PFS garantizado'
✅ Para qué SÍ sirve
  • TLS 1.3: único mecanismo de intercambio de claves (ECDHE obligatorio)
  • Garantizar PFS en VPNs y HTTPS
❌ Para qué NO sirve
  • ECDHE no cifra datos (establece la clave que luego usa AES-GCM)
  • No autentica identidades (eso es el certificado X.509)
🎯 Patrón Sec+

Pregunta sobre PFS o TLS 1.3 → ECDHE. Trampa: DHE (sin curva elíptica) también da PFS pero es más lento; ECDHE es el preferido moderno.

Hashing Cryptographic Hash Function ▼

Función unidireccional que convierte datos en un digest de tamaño fijo. MD5 (obsoleto), SHA-1 (obsoleto), SHA-256/SHA-3 (actuales). Detecta alteraciones.

🧠 Hash → 'Huella digital: mismo input, mismo output; cambio mínimo → hash totalmente diferente'
✅ Para qué SÍ sirve
  • Verificar integridad de archivos descargados (SHA-256 checksum)
  • Almacenar contraseñas (hash + salt, no cifrado)
❌ Para qué NO sirve
  • Cifrar datos (hash no es reversible, no se puede descifrar)
  • Garantizar confidencialidad (hash no oculta el contenido original a quien lo tenga)
🎯 Patrón Sec+

Pregunta: 'verificar que el archivo no fue alterado' → hash. 'guardar contraseñas' → hash+salt. Trampa: MD5/SHA-1 son colisión-vulnerables, no usar para seguridad.

PKI Public Key Infrastructure ▼

Infraestructura para emitir, gestionar y revocar certificados digitales. Componentes: CA raíz, CA intermedia, certificado X.509, CRL, OCSP.

🧠 PKI → 'La oficina de pasaportes de las claves públicas: valida identidades digitales'
✅ Para qué SÍ sirve
  • Emitir certificados TLS para HTTPS
  • Autenticación de usuarios con smart cards (LDAP + PKI)
❌ Para qué NO sirve
  • PKI no cifra datos directamente (los certificados facilitan el intercambio de claves)
  • Una CA comprometida invalida toda la confianza de la PKI
🎯 Patrón Sec+

Pregunta sobre 'confianza en certificados', 'CA', 'X.509' → PKI. Trampa: Certificate Pinning es una alternativa a confiar en la PKI completa (fija el certificado específico).

TLS Transport Layer Security ▼

Protocolo criptográfico para comunicaciones seguras (sucesor de SSL). TLS 1.3 es el estándar actual. Proporciona confidencialidad, integridad y autenticación del servidor.

🧠 TLS → 'El túnel seguro de Internet: ECDHE negocia, AES-GCM cifra, certificado autentica'
✅ Para qué SÍ sirve
  • HTTPS (TLS sobre HTTP)
  • Comunicación segura entre servicios internos (mTLS para autenticación mutua)
❌ Para qué NO sirve
  • TLS no garantiza que el servidor sea legítimo si confías en CAs maliciosas
  • TLS 1.0 y 1.1 están obsoletos y son vulnerables
🎯 Patrón Sec+

Preguntas sobre 'cifrado en tránsito', 'HTTPS', 'TLS handshake' → TLS. Trampa: TLS 1.3 eliminó ciphersuites débiles y RSA key exchange. mTLS ≠ TLS (mTLS autentica ambos lados).

Firma digital Digital Signature ▼

Mecanismo criptográfico que garantiza autenticidad, integridad y no-repudio. El emisor cifra un hash del mensaje con su clave privada. El receptor verifica con la clave pública.

🧠 Firma digital → 'Sello con clave privada: autentica quién firmó y que no fue alterado'
✅ Para qué SÍ sirve
  • Firmar código software (code signing) para verificar autor
  • Firmar emails (S/MIME) o documentos legales
❌ Para qué NO sirve
  • No garantiza confidencialidad (el mensaje puede ser legible)
  • No-repudio requiere que la clave privada esté bien custodiada
🎯 Patrón Sec+

Pregunta sobre 'autenticidad + integridad + no-repudio' → firma digital. Trampa: HMAC también da integridad pero no no-repudio (es simétrico, ambas partes tienen la misma clave).

HSM Hardware Security Module ▼

Dispositivo físico dedicado a la gestión segura de claves criptográficas. Protege claves privadas incluso de administradores del sistema. FIPS 140-2 Level 3+.

🧠 HSM → 'Caja fuerte física para claves: ni el admin puede sacar la clave privada'
✅ Para qué SÍ sirve
  • Almacenar claves privadas de CAs raíz
  • Firma de código en pipelines CI/CD seguros
❌ Para qué NO sirve
  • HSM no cifra datos directamente (los datos salen del HSM, solo la clave queda dentro)
  • No reemplaza KMS en la nube (aunque KMS puede estar respaldado por HSM)
🎯 Patrón Sec+

Pregunta: 'máxima protección para claves privadas de CA' → HSM. Trampa: TPM es similar pero integrado en el hardware del endpoint (no es un appliance dedicado).

TPMKMSPKIAES
TPM Trusted Platform Module ▼

Chip de seguridad integrado en la placa base. Almacena claves, mediciones de arranque (PCR) y secretos de forma segura. Base del arranque seguro (Secure Boot) y BitLocker.

🧠 TPM → 'Chip de confianza en la placa: protege claves de arranque y disco'
✅ Para qué SÍ sirve
  • BitLocker usa TPM para sellar la clave de cifrado de disco al estado del sistema
  • Attestation: probar que el sistema no fue alterado desde arranque
❌ Para qué NO sirve
  • TPM no cifra el disco completo por sí solo (BitLocker lo usa como custodio de la clave)
  • No reemplaza HSM para PKI empresarial
🎯 Patrón Sec+

Pregunta: 'cifrado de disco vinculado al hardware del equipo' → TPM+BitLocker. Trampa: si el TPM falla o la BIOS cambia, BitLocker pide la recovery key.

KMS Key Management System / Service ▼

Sistema centralizado para gestión del ciclo de vida de claves criptográficas: generación, distribución, rotación, revocación y almacenamiento. Ej: AWS KMS, Azure Key Vault.

🧠 KMS → 'El gestor centralizado de claves: ciclo de vida completo, auditable'
✅ Para qué SÍ sirve
  • Rotar claves de cifrado de base de datos automáticamente
  • Auditar quién usó qué clave cuándo (compliance)
❌ Para qué NO sirve
  • KMS no cifra los datos directamente (proporciona la clave para que la aplicación lo haga)
  • No reemplaza HSM cuando se requiere seguridad física dedicada (aunque KMS puede usar HSM internamente)
🎯 Patrón Sec+

Pregunta sobre 'gestión centralizada de claves en la nube' → KMS. Trampa: KMS ≠ HSM. KMS es el servicio/software; HSM es el hardware. AWS CloudHSM usa HSM físico.

Salt Cryptographic Salt ▼

Valor aleatorio único añadido a cada contraseña antes de hacer hash. Previene ataques de rainbow table y hace que hashes iguales sean diferentes.

🧠 Salt → 'Sal al hash: aunque dos usuarios tengan la misma contraseña, su hash es diferente'
✅ Para qué SÍ sirve
  • Almacenamiento seguro de contraseñas en bases de datos
  • Argon2id, bcrypt y scrypt incorporan salt automáticamente
❌ Para qué NO sirve
  • Salt no protege contraseñas débiles (solo aumenta el costo del ataque de fuerza bruta)
  • No es un secreto (el salt se almacena junto al hash, lo secreto es la contraseña)
🎯 Patrón Sec+

Pregunta: 'prevenir rainbow table attacks en almacenamiento de contraseñas' → salt. Trampa: el salt no es secreto; el pepper sí lo es (se almacena separado del hash).

Argon2id Argon2id Password Hashing ▼

Algoritmo de hashing de contraseñas ganador de la Password Hashing Competition (2015). Resistente a GPU y side-channel attacks. Recomendación OWASP para almacenamiento de contraseñas.

🧠 Argon2id → 'El mejor hash de contraseñas: lento a propósito, resistente a GPU'
✅ Para qué SÍ sirve
  • Almacenamiento de contraseñas en aplicaciones nuevas (reemplaza bcrypt)
  • Derivación de claves desde contraseñas (KDF)
❌ Para qué NO sirve
  • No para hash de datos en general (demasiado lento; SHA-256 para integridad)
  • Argon2d es vulnerable a side-channel; Argon2id es el balance recomendado
🎯 Patrón Sec+

Pregunta sobre 'mejor función para almacenar contraseñas' → Argon2id. Trampa: MD5 y SHA-1 son rápidos pero no adecuados para contraseñas.

CRL Certificate Revocation List ▼

Lista publicada por una CA con certificados revocados antes de su fecha de expiración. El cliente la descarga periódicamente para verificar si un certificado es válido.

🧠 CRL → 'Lista negra de certificados: descarga periódica, puede estar desactualizada'
✅ Para qué SÍ sirve
  • Revocar el certificado de un servidor comprometido
  • Verificar si un certificado de cliente sigue siendo válido
❌ Para qué NO sirve
  • CRL puede estar desactualizada (no es tiempo real como OCSP)
  • No todos los clientes verifican CRL (algunos ignoran la revocación)
🎯 Patrón Sec+

Pregunta sobre revocación de certificados: CRL (lista, periódica, puede estar cacheada) vs OCSP (en tiempo real). Trampa: OCSP Stapling mejora privacidad y rendimiento sobre OCSP básico.

OCSP Online Certificate Status Protocol ▼

Protocolo para verificar el estado de revocación de un certificado en tiempo real. OCSP Stapling: el servidor web incluye la respuesta OCSP en el handshake TLS.

🧠 OCSP → 'Consulta en tiempo real si el certificado sigue válido (vs CRL que es lista periódica)'
✅ Para qué SÍ sirve
  • Verificar revocación de certificados en tiempo real durante TLS handshake
  • OCSP Stapling para reducir latencia y mejorar privacidad
❌ Para qué NO sirve
  • OCSP básico expone al cliente ante la CA (privacidad). OCSP Stapling lo soluciona
  • OCSP no reemplaza la expiración de certificados
🎯 Patrón Sec+

Diferenciador clave: CRL = lista offline/batch, OCSP = consulta online/real-time. OCSP Stapling = el servidor pregunta a la CA y adjunta la respuesta al TLS handshake.

Propiedad criptográfica donde el compromiso de la clave privada del servidor no compromete sesiones pasadas. Se logra con claves de sesión efímeras (ECDHE).

🧠 PFS → 'Aunque roben la clave del servidor hoy, no pueden descifrar el tráfico de ayer'
✅ Para qué SÍ sirve
  • TLS 1.3 con ECDHE garantiza PFS obligatoriamente
  • VPNs con IKEv2 + ECDHE para proteger sesiones históricas
❌ Para qué NO sirve
  • RSA key exchange en TLS 1.2 NO da PFS (por eso TLS 1.3 lo eliminó)
  • PFS no protege el tráfico actual si la clave de sesión en uso es comprometida
🎯 Patrón Sec+

Pregunta: 'proteger sesiones pasadas aunque se comprometa la clave del servidor' → PFS / ECDHE. Trampa: TLS 1.2 con RSA key exchange no tiene PFS; TLS 1.3 siempre tiene PFS.

MITM Man-in-the-Middle Attack ▼

Types of Vulnerabilities

Memory Injection Memory Injection / Buffer Overflow ▼

Vulnerabilidades que permiten a un atacante escribir datos en zonas de memoria no autorizadas. Buffer overflow: escritura más allá del límite del buffer. Memory injection: inyección de código malicioso en el espacio de memoria de un proceso.

🧠 Buffer Overflow → "Desbordar el vaso: escribes más de lo que cabe y el exceso sobrescribe código de retorno"
✅ Para qué SÍ sirve
  • Explotar un buffer overflow en un servicio de red para ejecutar shellcode y obtener shell remoto
  • Comprender por qué ASLR, DEP/NX y stack canaries son mitigaciones clave en el examen
❌ Para qué NO sirve
  • ASLR no elimina buffer overflows, solo hace más difícil calcular las direcciones de memoria
  • No es exclusivo de C/C++: cualquier lenguaje que gestione memoria manualmente es susceptible
🎯 Patrón Sec+

Pregunta: "vulnerabilidad que permite ejecutar código arbitrario escribiendo más allá del buffer" → Buffer Overflow. Mitigaciones a recordar: ASLR (aleatoriza direcciones), DEP/NX (no ejecutar stack), Stack Canaries (detecta overflow).

Race Conditions Race Conditions — TOC/TOU ▼

Vulnerabilidad donde el resultado depende del orden de ejecución de procesos concurrentes. TOC (Time-of-Check): momento de verificar condición. TOU (Time-of-use): momento de usar el recurso. El atacante explota la ventana entre ambos.

🧠 TOC/TOU → "Entre que compruebas el billete y subes al tren, alguien te lo cambia: esa es la ventana de ataque"
✅ Para qué SÍ sirve
  • Explotar la ventana entre que se verifica un permiso y se ejecuta la acción (ej. symlink attacks)
  • Entender por qué transacciones atómicas y locks son la mitigación correcta
❌ Para qué NO sirve
  • Race conditions son difíciles de reproducir y detectar porque dependen del timing exacto
  • No afectan solo a sistemas multi-hilo: también a operaciones de sistema de ficheros
🎯 Patrón Sec+

Pregunta: "vulnerabilidad donde un atacante aprovecha la ventana entre verificación y uso" → Race Condition / TOCTOU. Mitigación: operaciones atómicas, file locking, transacciones de base de datos con ACID.

VM Escape Virtualization Vulnerabilities — VM Escape ▼

Vulnerabilidad que permite a un proceso dentro de una VM romper el aislamiento y acceder al hipervisor o a otras VMs. Resource reuse: datos de una VM quedan en memoria/disco y los lee otra VM.

  • VM Escape no es frecuente en producción pero es devastador si ocurre

Account Lockout Indicator — Account Lockout ▼

Indicador de posible ataque de fuerza bruta o password spraying: múltiples intentos fallidos de autenticación bloquean la cuenta. Analizar fuente de los intentos, horario y cuentas afectadas.

🧠 Account Lockout → "La alarma de la cerradura: demasiados intentos fallidos indican que alguien intenta entrar por la fuerza"

  • Detectar ataques de password spraying: muchas cuentas bloqueadas simultáneamente con pocos intentos cada una
  • Correlacionar en SIEM lockouts con IP de origen para identificar fuente del ataque
❌ Para qué NO sirve
  • Account lockout solo por sí mismo puede ser un ataque DoS contra cuentas (bloquear intencionalmente)
  • Lockout frecuente de una cuenta puede indicar usuario olvidadizo, no solo ataque
🎯 Patrón Sec+

Diferenciador: Brute force = muchos intentos en 1 cuenta → lockout rápido. Password spraying = 1-2 intentos en muchas cuentas → lockout más lento pero difícil de detectar. Trampa: lockout ≠ siempre ataque.

Mitigation Techniques

Segmentation Network Segmentation ▼

División de la red en zonas aisladas (VLANs, subnets, DMZ) para limitar el movimiento lateral de un atacante. Si una zona es comprometida, las demás permanecen protegidas.

🧠 Segmentation → 'Compartimentos estancos: un barco no se hunde entero si solo entra agua en una sala'
✅ Para qué SÍ sirve
  • Aislar sistemas OT/SCADA de la red corporativa
  • Contener el movimiento lateral tras un ransomware
❌ Para qué NO sirve
  • Segmentación mal configurada (ACLs permisivas) no aporta protección real
  • No protege contra amenazas internas que ya tienen acceso a múltiples segmentos
🎯 Patrón Sec+

Pregunta: 'limitar el impacto si un sistema es comprometido' → Segmentation. Trampa: segmentación física (air-gap) vs lógica (VLAN). VLAN puede saltar con VLAN hopping si no se configura bien.

Access Control List ACL — Access Control List ▼

Lista de reglas que define qué tráfico de red o qué accesos a recursos están permitidos o denegados. Puede ser a nivel de red (router/firewall) o de sistema de ficheros.

🧠 ACL → 'La lista de invitados: solo entra quien está en la lista y cumple las condiciones'
✅ Para qué SÍ sirve
  • Filtrar tráfico entrante/saliente en routers y firewalls por IP, puerto y protocolo
  • Controlar permisos de acceso a ficheros y directorios (DACL en Windows)
❌ Para qué NO sirve
  • ACL de red no inspecciona el contenido del paquete (eso es IDS/IPS o WAF)
  • ACL demasiado permisiva es peor que no tenerla (falsa sensación de seguridad)
🎯 Patrón Sec+

Proceso de identificar, adquirir, probar e instalar parches de software y firmware. Reduce la ventana de exposición a vulnerabilidades conocidas. Crítico para CVEs con exploit público.

🧠 Patch Management → 'Cerrar las ventanas abiertas antes de que entre el ladrón: parchea rápido lo crítico'
✅ Para qué SÍ sirve
  • Reducir el riesgo de explotación de CVEs conocidos (EternalBlue, Log4Shell)
  • Cumplimiento PCI-DSS: parches críticos en menos de 30 días
❌ Para qué NO sirve
  • No protege contra zero-days (vulnerabilidades sin parche disponible)
  • Parchear sin probar primero puede romper aplicaciones en producción

Escenario: 'sistema comprometido por vulnerabilidad conocida hace 6 meses' → fallo de Patch Management. Compensating control cuando no se puede parchear: segmentación + IPS + monitorización extra.

Resilience & Recovery

BCP/DRP Business Continuity Plan / Disaster Recovery Plan ▼

BCP: plan para mantener funciones críticas durante un desastre. DRP: plan técnico para recuperar sistemas IT. BCP es más amplio (negocio); DRP es el componente técnico.

🧠 BCP/DRP → 'BCP mantiene el negocio vivo, DRP recupera los sistemas: el mapa del caos'
✅ Para qué SÍ sirve
  • Definir procedimientos ante ransomware, terremoto o fallo de datacenter
  • Documentar RTO/RPO para priorizar la recuperación de sistemas críticos
❌ Para qué NO sirve
  • BCP no es solo IT (incluye operaciones, comunicaciones, personas)
  • DRP sin pruebas periódicas no vale: si no lo has probado, no funciona en un desastre real
🎯 Patrón Sec+

Pregunta: 'plan para mantener el negocio durante un incidente mayor' → BCP. 'plan técnico para recuperar sistemas' → DRP. Los dos se mencionan juntos en el examen.

RTO/RPO Recovery Time Objective / Recovery Point Objective ▼

RTO: tiempo máximo tolerable para restaurar el servicio. RPO: máxima pérdida de datos tolerable (definida en tiempo desde el último backup). Guían la arquitectura de recuperación.

🧠 RTO/RPO → 'RTO: cuánto tiempo puedes estar caído. RPO: cuántos datos puedes perder'
✅ Para qué SÍ sirve
  • Definir frecuencia de backups basada en RPO (RPO=1h → backup cada hora)
  • Justificar inversión en failover activo-activo vs activo-pasivo según RTO
❌ Para qué NO sirve
  • RTO bajo no implica RPO bajo (son independientes aunque relacionados)
  • Backups diarios con RPO=1h no tienen sentido (el backup es menos frecuente que el RPO)
🎯 Patrón Sec+

Regla: RTO = downtime aceptable → define el tipo de HA. RPO = pérdida de datos aceptable → define frecuencia de backup/replicación. Trampa: confundir los dos.

Hot/Warm/Cold Site Disaster Recovery Sites ▼

Tipos de sitios de recuperación ante desastres. Hot site: réplica completa activa (RTO minutos). Warm site: infraestructura parcial preconfigurada (RTO horas). Cold site: solo espacio físico (RTO días/semanas).

🧠 Hot/Warm/Cold → "Hot = gemelo encendido. Warm = gemelo dormido. Cold = habitación vacía esperando muebles"
✅ Para qué SÍ sirve
  • Hot site para sistemas críticos con RTO < 1 hora: banca, hospitales, infraestructura crítica
  • Cold site cuando el coste de hot/warm site es prohibitivo y el RTO puede ser de días
❌ Para qué NO sirve
  • Hot site es 3-5x más caro que cold site: no todo necesita hot site, clasificar por criticidad
  • Warm site requiere tiempo para configurar y cargar datos: validar que el RTO es alcanzable
🎯 Patrón Sec+

Pregunta: "sistema crítico debe recuperarse en menos de 1 hora" → Hot Site. Diferenciadores de coste: Hot (mayor coste, menor RTO) > Warm > Cold (menor coste, mayor RTO). Geographic dispersion = sitio en región diferente para resiliencia ante desastres regionales.

Full: copia completa de todos los datos. Incremental: solo cambios desde el último backup (full o incremental). Differential: cambios desde el último full backup. Cada tipo tiene diferente balance entre tiempo de backup y tiempo de restauración.

DMARC DMARC — Domain-based Message Authentication Reporting and Conformance ▼

Protocolo de autenticación de email que combina SPF y DKIM. Define qué hacer con emails que fallan la verificación: none (monitorizar), quarantine (spam), reject (bloquear). Protege contra phishing y spoofing del dominio.

🧠 DMARC → "El portero del email: si el remitente no está en la lista (SPF) o la firma es falsa (DKIM), a la basura"
✅ Para qué SÍ sirve
  • Proteger el dominio corporativo para que nadie pueda enviar emails haciéndose pasar por tu empresa
  • Política p=reject: los emails que fallan SPF+DKIM son rechazados directamente por el servidor receptor
❌ Para qué NO sirve
  • DMARC en modo "none" solo monitoriza, no protege — necesitas pasar a quarantine/reject para proteger
  • DMARC no cifra el contenido del email: protege la autenticidad del remitente, no la confidencialidad
🎯 Patrón Sec+

Orden de implementación: SPF (lista IPs autorizadas) → DKIM (firma criptográfica del email) → DMARC (política de qué hacer si fallan). Pregunta: "evitar que atacantes envíen emails suplantando tu dominio" → DMARC p=reject.

SPF SPF — Sender Policy Framework ▼

Registro DNS que lista los servidores de correo autorizados a enviar email en nombre del dominio. El servidor receptor verifica que el email proviene de una IP listada en el registro SPF. Primera línea de defensa anti-spoofing.

🧠 SPF → "La lista de IPs permitidas en el DNS: si el email viene de otra IP, es sospechoso"
✅ Para qué SÍ sirve
  • Publicar registro SPF en DNS para todos los dominios que envían email (incluyendo subdominios)
  • Combinar con DKIM y DMARC para protección completa contra email spoofing
❌ Para qué NO sirve
  • SPF solo no es suficiente: un atacante puede enviar desde un servidor autorizado si lo compromete
  • SPF tiene límite de 10 lookups DNS: registros complejos pueden fallar la validación
🎯 Patrón Sec+

Pregunta sobre el orden de las capas de email security: SPF (IP del servidor) + DKIM (firma del mensaje) + DMARC (política de enforcement). SPF valida "¿desde qué IP?" y DKIM valida "¿está el contenido firmado por el dominio?".

MAC MAC — Mandatory Access Control ▼

Modelo de control de acceso donde el SO asigna etiquetas de seguridad a sujetos (usuarios) y objetos (ficheros). El acceso se decide comparando etiquetas según una política central. Nadie, ni el propietario, puede cambiar las reglas. Usado en entornos militares y gobierno.

🧠 MAC → "El control de acceso del ejército: la etiqueta de clasificación lo decide todo, ni el dueño puede cambiarlo"

  • SELinux y AppArmor implementan MAC en Linux: el SO decide qué puede hacer cada proceso
❌ Para qué NO sirve
  • MAC es muy rígido: difícil de administrar en entornos comerciales con necesidades de acceso dinámicas
  • No confundir MAC (Mandatory) con MAC (Media Access Control) de direcciones de red
🎯 Patrón Sec+

Diferenciadores de modelos de control de acceso: MAC (etiquetas, control central, más rígido) → DAC (propietario decide, más flexible, ej. NTFS) → RBAC (roles) → ABAC (atributos). Pregunta sobre "control más estricto" → MAC.

DAC DAC — Discretionary Access Control ▼

NIST IR NIST Incident Response Framework (SP 800-61) ▼

Framework de respuesta a incidentes con 4 fases: Preparation → Detection & Analysis → Containment, Eradication & Recovery → Post-Incident Activity (lecciones aprendidas).

🧠 NIST IR → 'El ciclo del bombero: preparar, detectar, contener/erradicar/recuperar, aprender'

  • Post-incident review para mejorar controles y no repetir el incidente
❌ Para qué NO sirve
  • NIST IR no prescribe herramientas específicas (es un framework de proceso)
  • Post-incident es la fase más olvidada pero crítica para la mejora continua
🎯 Patrón Sec+

Orden de fases NIST IR: Prep → Detection → Containment → Eradication → Recovery → Lessons Learned. Trampa: el examen pregunta el orden correcto. 'Lessons Learned' siempre es la última fase.

Log Analysis Log Analysis & Data Sources ▼

🧠 Log Analysis → 'Los logs son el rastro del crimen: sin ellos, la investigación es ciega'

NetFlow NetFlow / Network Traffic Analysis ▼

Protocolo de Cisco para recopilar metadatos de flujos de red (IP origen/destino, puertos, duración, bytes transferidos). No captura el contenido del paquete, solo metadatos de flujo. Útil para detectar C2, exfiltración y movimiento lateral.

🧠 NetFlow → 'El registro de llamadas de la red: quién habló con quién, cuándo y cuánto, sin oír el contenido'
✅ Para qué SÍ sirve
  • Detectar exfiltración masiva de datos por volumen inusual a IP externa
  • Identificar conexiones a C2 por patrones de beaconing (tráfico regular a IP sospechosa)

Privacy Privacy — Data Subject Rights & Controller/Processor ▼

GDPR y marcos de privacidad modernos establecen derechos de los interesados (data subjects): acceso, rectificación, supresión (right to be forgotten), portabilidad. Controller = decide propósito del tratamiento. Processor = trata datos por cuenta del controller.

🧠 Privacy → "El GDPR da poder al ciudadano sobre sus datos: derecho a saber, corregir, borrar y llevárselos"
✅ Para qué SÍ sirve
  • Right to be forgotten: el usuario solicita borrado de sus datos personales y la empresa debe cumplir (con excepciones legales)
  • Data inventory + retention policy: saber qué datos tienes, dónde están y cuánto tiempo los guardas
❌ Para qué NO sirve
  • Right to be forgotten no es absoluto: datos necesarios para cumplimiento legal no pueden borrarse
  • Controller puede ser responsable de brechas del Processor: el contrato debe incluir cláusulas de seguridad (DPA)
🎯 Patrón Sec+

Diferenciadores GDPR: Data Subject (persona cuyos datos se tratan) → Controller (decide qué y para qué) → Processor (trata datos por cuenta del controller). Data sovereignty = los datos deben residir en el país/región donde aplica la ley.

Non-compliance Consequences Consequences of Non-compliance ▼

Consecuencias de no cumplir las regulaciones de seguridad: multas económicas (GDPR hasta 4% ingresos globales), sanciones regulatorias, daño reputacional, pérdida de licencias para operar, impactos contractuales con clientes y partners.

Audits & Assessments

Penetration Testing Penetration Testing ▼

Ataque simulado y autorizado contra sistemas para identificar vulnerabilidades explotables antes que un atacante real. Tipos: Black box (sin info), White box (info completa), Grey box (info parcial). Entornos: Known, Partially Known, Unknown.

🧠 Pentest → 'Hackéate a ti mismo primero: mejor que te lo encuentre tu equipo que el atacante'
✅ Para qué SÍ sirve
  • Validar que los controles existentes resisten un ataque real, no solo teórico
  • Cumplimiento PCI-DSS, SOC2: requieren pentest anual o tras cambios significativos
❌ Para qué NO sirve
  • Pentest puntual no garantiza seguridad continua (el entorno cambia constantemente)
  • No reemplaza vulnerability scanning continuo ni monitorización permanente
🎯 Patrón Sec+

Diferenciadores clave: Black box = Unknown environment (simula atacante externo). White box = Known environment (máxima info, más eficiente). Reconocimiento: Pasivo (sin interacción, OSINT) vs Activo (escaneo directo, detectable).

Security Audit Security Audit (Internal / External) ▼

Evaluación formal de controles, políticas y procedimientos de seguridad contra un estándar o framework. Interna (por el propio equipo o auditor interno) o externa (tercero independiente). Produce evidencia de cumplimiento.

🧠 Security Audit → 'La inspección oficial: ¿tus controles cumplen lo que dicen cumplir?'

  • Identificar desviaciones entre políticas documentadas y práctica real
❌ Para qué NO sirve
  • Auditoría puntual no detecta vulnerabilidades técnicas explotables (eso es pentest/vuln scan)
  • Audit committee interno puede tener sesgo; auditores externos son más objetivos
