D1 · General RPO vs RTO 🔥 MUY FALLADO
RPORTO
CompletoRecovery Point ObjectiveRecovery Time Objective
Pregunta¿Cuántos datos puedo perder?¿Cuánto puedo estar caído?
Dirección← hacia el PASADO→ hacia el FUTURO
ControlaFrecuencia de backupsVelocidad de restauración
RPO = 1h → pérdida tolerable ≤ 60min · Pierdes 45min → 45 < 60 → RPO NO violado ✅
🔑 La lógica exacta
RPO = umbral máximo de pérdida. Pérdida < RPO = dentro del margen (OK).
Se viola solo si la pérdida supera el RPO. Ejemplo: backup cada hora falla dos veces → pierdes 1h45min → viola RPO de 1h.
Caso 45min con RPO=1h → NO viola (algunos simulacros tienen esta pregunta mal)
⚡ Nemotecnia
"Point = Pasado" (datos perdidos) · "Time = Tiempo de recuperación" (hacia adelante)
RPO es el máximo tolerable — si pierdes MENOS que el RPO, estás dentro del límite.

D1 · General Kerberos — Componentes y Flujo 🔥 MUY FALLADO
ComponenteFunción
KDCKey Distribution Center — contiene AS + TGS
ASAuthentication Server — emite el TGT inicial
TGSTicket Granting Server — emite tickets de servicio (TGS)
TGTTicket Granting Ticket — "pase" para pedir más tickets
1. AS-REQ → KDC/AS → TGT (cifrado con KRBTGT) 2. TGS-REQ → KDC/TGS → TGS ticket (cifrado con hash cuenta servicio) 3. AP-REQ → Servicio → acceso
🔑 Confusiones comunes
TGT ≠ componente que lo emite. El AS emite el TGT.
TGS = servidor Y ticket (contexto distingue). El TGS emite tickets de servicio.
⚡ Nemotecnia
AS → TGT → TGS → servicio
"El AS te da el pase (TGT); con el pase, el TGS te da la entrada al concierto"

D1 · General HMAC vs Hash Simple 🔥 MUY FALLADO
Hash simple (SHA-256): verifica integridad, pero cualquiera puede recalcular el hash de un mensaje modificado.

HMAC: incorpora una clave secreta compartida en el cálculo. Solo quien conoce la clave puede generar/verificar el HMAC → garantiza integridad + autenticidad.
HMAC = H(clave_secreta ‖ mensaje)
🔑 Clave examen
"Verificar que el mensaje no ha sido alterado y proviene del remitente" → HMAC
"Solo integridad" → hash simple. HMAC NO cifra; solo autentica.

D1 · General NTLM Challenge-Response
NTLM usa un mecanismo de desafío-respuesta basado en hash sin transmitir la contraseña en texto plano.

Flujo: servidor envía nonce (challenge) → cliente responde con H(hash_NTLM + nonce) → servidor verifica.
🔑 Confusión con otros
Autenticación básica HTTP = envía contraseña en Base64 (texto plano)
Kerberos = usa tickets cifrados (no challenge-response directo)
OAuth = autorización delegada (no autenticación directa)

D2 · Amenazas XSS — Stored vs Reflected vs DOM-based 🔥 MUY FALLADO
TipoCómo funcionaSeñal en pregunta
StoredPayload se guarda en BD, se ejecuta en cada visita"guarda en BD", "todos los visitantes lo ven"

D3 · Arquitectura DMZ con Dos Firewalls
Internet → [FW1] → DMZ (servidores públicos) → [FW2] → LAN interna
Si FW1 es comprometido, el atacante accede a la DMZ pero FW2 sigue protegiendo la LAN interna. Con un solo firewall, comprometer el FW = acceso total.
🔑 Principio subyacente
Defense in Depth — múltiples capas de seguridad independientes.
⚡ Nemotecnia
"Un firewall caído ≠ red interna caída si hay dos"

D3 · Arquitectura TLS en HTTPS — Qué hace cada algoritmo
ElementoRol en HTTPS
TLSEl protocolo completo que gestiona todo el proceso de cifrado en HTTPS
RSA / ECCAsimétrico — usado en el handshake para intercambiar claves de sesión
AESSimétrico — cifra los datos reales (rápido, usado durante la sesión)
SHA-256Hash — verifica integridad de los datos, NO cifra nada
🔑 Clave examen

TLS es el contenedor — RSA/AES/SHA son las piezas que usa internamente
